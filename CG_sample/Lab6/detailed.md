# Лаб 6 «Система частиц» — подробный разбор для защиты

Стек: **WebGL1** + JS, всё обёрнуто в IIFE `(() => { ... })()`. Файл — [main.js](main.js).

Главная задача: реализовать **систему частиц** с 8 разными эффектами (бенгальский огонь, дым, дождь, облака+пар, фейерверки, северное сияние, снег, бенгальский огонь «лекционный» с лучами). До 18 000 частиц одновременно, всё рисуется через `gl.POINTS` в один draw call на эффект.

---

## 0. Что такое «система частиц»

Архитектурный паттерн: вместо моделирования одного сложного объекта (огонь, дым) рисуем **тысячи мелких самостоятельных «штук»** — частиц, каждая из которых:
- Имеет позицию, скорость, ускорение, время жизни.
- Управляется простой физикой (`v += a·dt; pos += v·dt`).
- Рождается в **эмиттере** (источнике) и **умирает** по истечении `life`.
- Параметры (размер, цвет, прозрачность) **анимируются** от начальных к конечным.

В сумме это даёт правдоподобный «непрерывный» эффект — пламя, дождь, дым — без реального моделирования жидкости/газа.

В WebGL частицы обычно рисуются режимом `gl.POINTS`: каждая вершина = одна частица = один квадратный спрайт, размер которого задаётся в шейдере через `gl_PointSize`.

---

## 1. Инициализация (main.js:1–8)

```js
const gl = canvas.getContext('webgl', { alpha: false, antialias: true });
```
- `alpha: false` — у канваса нет альфа-канала, что чуть быстрее.
- `antialias: true` — MSAA для геометрии (на самих точках не сказывается).

## 2. Шейдер частиц (main.js:9–65)

### Вершинный
```glsl
attribute vec2 a_position;   // позиция в пикселях
attribute float a_size;      // размер точки
attribute vec4 a_color;      // цвет + альфа
attribute float a_sprite;    // тип спрайта (0 круг, 1 бенгал, 2 снег)

uniform vec2 u_resolution;   // размер канваса

void main() {
  vec2 zeroToOne = a_position / u_resolution;
  vec2 clipSpace = zeroToOne * 2.0 - 1.0;
  gl_Position = vec4(clipSpace * vec2(1.0, -1.0), 0.0, 1.0);  // флип Y
  gl_PointSize = a_size;
  v_color = a_color;
  v_sprite = a_sprite;
}
```

Самые важные вещи:
- **Координаты в пикселях** → clip-space `[-1, 1]`. Делим на разрешение, масштабируем ×2, сдвигаем −1, флипаем Y (в WebGL +Y вверх, в экранных координатах +Y вниз).
- **`gl_PointSize`** — встроенный output, размер квадратного спрайта в пикселях. Только в режиме `POINTS`. Каждая частица может иметь свой.

### Фрагментный — суть процедурного спрайта
```glsl
vec2 p = gl_PointCoord - vec2(0.5);
float d = length(p) * 2.0;                          // 0 в центре, 1 на краю
float core = smoothstep(0.75, 0.0, d);              // плотное ядро
float glow = smoothstep(1.0, 0.2, d) * 0.35;        // мягкое halo
float radialAlpha = (core + glow) * v_color.a;
```

`gl_PointCoord` — встроенная переменная **только для POINTS-режима**, диапазон `[0,1] × [0,1]` внутри спрайта-точки. Позволяет рисовать произвольный паттерн (круг, текстура) на каждой точке без отдельной геометрии.

`smoothstep(a, b, x)` — плавная интерполяция между 0 и 1 через гермитов сплайн. `smoothstep(0.75, 0, d)` означает: при d=0 (центр) → 1, при d=0.75 → 0, между — мягко падает. Так получается **круглый блик с halo** прямо из квадратной точки.

Если у частицы выставлен `spriteType`, вместо процедурного блика читается реальная PNG-текстура:
```glsl
if (spriteType == 1 && u_bengReady > 0.5) {
  vec4 sprite = texture2D(u_bengTexture, gl_PointCoord);
  gl_FragColor = vec4(v_color.rgb * sprite.rgb, v_color.a * sprite.a);
}
```
Цвет частицы умножается на цвет текстуры — тонирование (например, бенгальский огонь подкрашивается оранжевым).

## 3. Шейдер «следов» (main.js:67–93)

Отдельная простая программа `trailProgram` для режима **Q** (Sparkler Lecture). Просто рисует цветные линии (`gl.LINES`) — от точки эмиссии до текущей позиции частицы. Используется атрибут `a_position` + `a_color`, без всякого блендинга.

## 4. Вспомогательные функции (main.js:127–164)

- **`randomRange(min, max)`** — равномерное число в `[min, max)`.
- **`lerp(a, b, t)`** — линейная интерполяция `a + (b−a)·t`. Основа анимации параметров частицы.
- **`emissionCount(rate, dt)`** — ожидаемое количество частиц за `dt`: целая часть + стохастический «остаток» (с вероятностью дробной части добавляется ещё одна). Это даёт корректную среднюю частоту эмиссии независимо от FPS.
- **`gradientColor(u)`** — многоточечный цветовой градиент с 4 опорными цветами (циан→синий→фиолетовый→розовый) для северного сияния. Циклится по `(u % 1)`.

## 5. Текстуры (main.js:166–200)

`createTexturePlaceholder()` — создаёт текстуру 1×1 (белый пиксель), чтобы рендер работал сразу. `loadTexture()` асинхронно подгружает PNG и заменяет содержимое. Флаги `sparklerTextureReady` / `snowTextureReady` передаются в шейдер как `u_bengReady` / `u_snowReady` — пока флаг = 0, шейдер использует процедурный круг.

**`UNPACK_PREMULTIPLY_ALPHA_WEBGL = true`** — RGB умножается на A при загрузке. Это нужно для корректного смешивания: без премультипликации полупрозрачные края спрайта дают «тёмный ореол».

## 6. Глобальное состояние (main.js:263–296)

```js
const gravity = 450;             // пикс/сек² — глобальная гравитация
const particles = [];            // пул живых частиц
const rockets = [];              // отдельный пул для фейерверков
const state = {
  mode: 1,
  maxParticles: 18000,
  sizeScale: 1.7,                // глобальный множитель размера
  ...
};
```

`setMode(mode)` чистит пулы и меняет режим, обновляет `document.title`.

Клавиши **1–6** — режимы 1–6, **E** — снег (mode 7), **Q** — sparkler lecture (mode 8). Обрабатывается и кириллическая раскладка (`у`, `й`).

## 7. `spawnParticle(p)` (main.js:315) — фабрика частицы

Заталкивает в массив `particles` объект с полями:
- `x, y` — текущая позиция; `originX, originY` — точка рождения (для следов в режиме Q); `prevX, prevY` — для интерполяций.
- `vx, vy` — скорость; `ax, ay` — ускорение; `drag` — сопротивление среды (1/сек).
- `life, maxLife` — таймер.
- `size → sizeEnd` — анимация размера; `color → colorEnd` — цвета; `alpha → alphaEnd` — прозрачности.
- `fadeIn` — время плавного появления.
- `gravityScale` — множитель глобальной гравитации (0 для пара/огня, отрицательный для дыма/облаков «всплывают», 1 для типичных).
- `sprite` — 0/1/2 (тип спрайта).
- `kind` — строковая метка, влияет на спец-логику (`sparkler` добавляет шлейф из мини-искр).

## 8. Эмиттеры эффектов (main.js:348–675)

Каждый режим — отдельная функция `spawn*(dt, t)`. Пример паттерна для **бенгальского огня**:

```js
function spawnSparkler(dt, t) {
  const count = emissionCount(340, dt);            // ~340 частиц/сек
  const x = width * 0.5, y = height * 0.58;
  for (let i = 0; i < count; i++) {
    const ang = randomRange(0, 2π) + sin(t·9 + i·0.2) * 0.12;  // лёгкое колыхание
    const speed = randomRange(180, 360);
    spawnParticle({
      x, y,
      vx: cos(ang)·speed + jitter,
      vy: sin(ang)·speed,
      drag: 1.1,                  // быстро тормозит
      life: rand(1.0, 1.9),
      size: rand(6.8, 14.8),
      sizeEnd: 1,                 // уменьшается до точки
      color: [1, ~0.85, ~0.35],   // тёплый жёлто-оранжевый
      colorEnd: [1, 0.2, 0.05],   // догорает в красный
      alpha: 0.95, alphaEnd: 0,
      gravityScale: 0.48,         // немного падают
      spriteType: SPRITE_BENG,
      kind: 'sparkler'
    });
  }
}
```

Остальные режимы — вариации того же приёма:
- **Smoke** — большие серые частицы, `gravityScale: -0.16` (всплывают), `size → sizeEnd` сильно растёт.
- **Rain** — много быстрых маленьких сине-голубых точек по всей ширине, `vy ≈ 330–580`.
- **Clouds + Steam** — два эмиттера: облака сверху (кластерами, очень долгая жизнь, fadeIn) + 6 «струй» пара снизу с колебанием `sin(t·2 + j)`.
- **Fireworks** — отдельная логика с ракетами (см. ниже).
- **Northen Lights** — 3 «полосы» (lanes), вертикальное колебание `sin/cos`, цвет из `gradientColor()`, очень полупрозрачные крупные частицы.
- **Snow** — снежинки PNG-спрайтом, ветер `sin(t·0.7 + y·0.04)·35`.
- **Sparkler Lecture** — то же, что sparkler, но с `kind: 'sparklerLecture'` и сохранённым `originX/Y` — для отрисовки лучей от центра.

## 9. Фейерверки (main.js:526–635) — ракеты + взрывы

Двухэтапный эффект:

1. **Ракета** (`rockets[]`) — отдельный объект, не частица. Летит вверх с уменьшенной гравитацией (`gravity * 0.34`), каждый кадр оставляет короткоживущие «искры хвоста» через обычный `spawnParticle`. Когда `vy >= -20` (почти остановилась) или достигнута `targetY` — вызывается `explodeFirework(x, y, color, type)` и ракета удаляется.

2. **`explodeFirework`** — спавнит 110–170 частиц одним залпом. **4 типа взрыва**:
   - **0** (случайный шар) — равномерно случайные углы и скорости.
   - **1** (ровное кольцо) — все скорости близки к одной → ровный круг.
   - **2** (спираль) — угол `(2π·i)/burstCount` + лёгкий jitter → ровно распределённые лучи.
   - **3** (цветок) — `speed *= sin(ang · 6)` → 6 лепестков.

Плюс 34 серых «дымных» частицы поверх для атмосферности.

## 10. `updateParticles(dt)` (main.js:684) — физика

Для каждой частицы (сзади наперёд, чтобы безопасно удалять):
```js
p.life -= dt;
if (p.life <= 0) { splice(i, 1); continue; }

// Drag (сопротивление)
const dragFactor = max(0, 1 − p.drag · dt);
p.vx *= dragFactor;
p.vy *= dragFactor;

// Ускорение + гравитация
p.vx += p.ax · dt;
p.vy += (p.ay + gravity · p.gravityScale) · dt;

// Позиция
p.x += p.vx · dt;
p.y += p.vy · dt;
```

Это **полуэвклидов интегратор** (semi-implicit Euler) — простой, устойчивый, годится для эффектов, не симуляции физики. Особенность лабы: для sparkler-частиц в `update` ещё спавнятся короткоживущие «trail»-искры — даёт видимый шлейф из искорок.

## 11. `fillBuffers()` (main.js:~750) — упаковка для GPU

Раз в кадр обходит весь массив и для каждой живой частицы записывает:
```js
const t = 1 - p.life / p.maxLife;            // от 0 (рождение) до 1 (смерть)
let alpha = lerp(p.alpha, p.alphaEnd, t);
if (p.fadeIn > 0 && (maxLife − life) < fadeIn)
  alpha *= (maxLife − life) / fadeIn;         // плавный fade-in

positions[i*2]   = p.x;  positions[i*2+1] = p.y;
sizes[i]         = lerp(p.size, p.sizeEnd, t);
colors[i*4..+3]  = lerp(p.color, p.colorEnd, t) + alpha;
sprites[i]       = p.sprite;
```

Это превращает массив объектов в плотные типизированные массивы (`Float32Array`) — то, что GPU умеет читать одним блоком. Ограничение `state.maxParticles = 18000` срезает лишние, не давая массиву расти бесконечно.

## 12. `draw(count, trailCount)` (main.js:801)

1. Очистка `COLOR_BUFFER_BIT` (depth не используется — частицы всегда поверх друг друга).
2. **Блендинг**: `gl.enable(BLEND)` + `gl.blendFunc(SRC_ALPHA, ONE_MINUS_SRC_ALPHA)` — стандартная альфа-композиция: цвет на экране = `src·srcAlpha + dst·(1−srcAlpha)`. Для частиц с премультипликацией строго правильнее `ONE, ONE_MINUS_SRC_ALPHA`, но здесь используется классический вариант.
3. Если нужны trail'ы — переключиться на `trailProgram`, залить и нарисовать через `drawArrays(gl.LINES, 0, n)`.
4. Переключиться на `particleProgram`, залить uniform'ы и текстуры в юниты 0/1.
5. Для каждого VBO: `bindBuffer → bufferData(DYNAMIC_DRAW)` — заливаем свежие данные каждый кадр → `vertexAttribPointer`.
6. **Один** `gl.drawArrays(gl.POINTS, 0, count)` рисует все частицы.

`DYNAMIC_DRAW` — подсказка драйверу: «эти данные меняются каждый кадр», в отличие от `STATIC_DRAW` для геометрии моделей.

## 13. Главный цикл `frame(now)` (main.js:897)

```js
function frame(now) {
  resize();                                  // DPI-aware
  const dt = min(0.033, (now − lastTime) * 0.001);   // ограничение dt сверху
  lastTime = now;
  state.time += dt;

  spawnByMode(dt, state.time);               // эмиттер активного режима
  updateParticles(dt);                       // физика всех частиц
  const count = fillBuffers();               // упаковка для GPU
  const trailVertexCount = state.mode === 8 ? fillTrailBuffers() : 0;
  draw(count, trailVertexCount);
  requestAnimationFrame(frame);
}
```

`dt` ограничивается `33мс` (≈30 FPS), чтобы при заминке (alt-tab) частицы не «улетели в космос» за один шаг.

## Что точно спросят на защите

| Вопрос | Ответ |
|---|---|
| Что такое система частиц? | Архитектурный паттерн: тысячи мелких независимых объектов, каждый со своей физикой и временем жизни, рождаются в эмиттере, умирают по таймеру. Имитируют непрерывные процессы (огонь, дым, дождь). |
| Почему `gl.POINTS`, а не quad-ы? | Один POINT = одна вершина = один спрайт. Экономия памяти и драй-коллов: 18 000 частиц = 18 000 вершин, а не 72 000 (по 4 на quad). Драйвер сам растеризует квадратный спрайт указанного размера. |
| Что такое `gl_PointSize`? | Встроенный output вершинного шейдера, размер квадратного спрайта точки в пикселях. Только в режиме `POINTS`. |
| Что такое `gl_PointCoord`? | Встроенная переменная фрагментного шейдера в POINTS-режиме: UV-координаты `[0,1]²` внутри спрайта-точки. Позволяет нарисовать круг, текстуру, любой паттерн. |
| Как из квадратной точки получить круглую частицу? | `d = length(gl_PointCoord − 0.5) * 2`, потом `smoothstep` для мягкого края. Прозрачность за пределами «круга» = 0. |
| Что такое `smoothstep(a, b, x)`? | Плавная (S-образная) интерполяция между 0 и 1 на интервале `[a, b]`. Используется для мягких границ вместо резкого `step`. |
| Как работает блендинг и зачем `blendFunc(SRC_ALPHA, ONE_MINUS_SRC_ALPHA)`? | Включение `gl.BLEND` заставляет GPU смешивать src (новый пиксель) с dst (тем, что уже на экране) по формуле `src·a + dst·(1−a)`. Без блендинга прозрачные частицы рисовались бы прямоугольниками. |
| Зачем `UNPACK_PREMULTIPLY_ALPHA_WEBGL = true`? | Заранее умножает RGB на A при загрузке текстуры. Иначе на полупрозрачных краях спрайта получится тёмный ореол из-за фильтрации. |
| Как реализована физика частиц? | Полуэвклидов Эйлер: `v += a·dt; pos += v·dt`. Drag через `v *= (1 − drag·dt)`. Гравитация — глобальная константа, умножается на `gravityScale` каждой частицы. |
| Зачем `gravityScale` отрицательный? | Для дыма и облаков — они «всплывают», то есть гравитация направлена вверх. |
| Как работает анимация цвета/размера во времени? | `t = 1 − life/maxLife` (от 0 в рождение до 1 в смерть), потом `lerp(start, end, t)` для каждого параметра. |
| Что такое `emissionCount(rate, dt)`? | Расчёт количества частиц за кадр с поправкой на дробный остаток через rng — чтобы средняя частота эмиссии не зависела от FPS. |
| Зачем `DYNAMIC_DRAW` вместо `STATIC_DRAW`? | Это подсказка драйверу: данные меняются каждый кадр. STATIC_DRAW оптимизирован под «загрузил и не трогаю», DYNAMIC — под частые перезаливки. |
| Почему сначала пишутся в Float32Array, а потом в GPU? | Потому что GPU читает плотные типизированные массивы одним блоком. Массив JS-объектов GPU прочитать не может — сначала надо «развернуть» поля в плотные буферы. |
| Чем 4 типа взрыва фейерверка отличаются? | 0: случайный шар. 1: ровное кольцо (одинаковые скорости). 2: спираль (углы равномерно). 3: «цветок» — скорость модулируется `sin(angle·6)`, даёт 6 лепестков. |
| Почему ракеты в отдельном массиве `rockets[]`, а не в `particles[]`? | У них своя логика (отдельный update, проверка достижения `targetY`, вызов `explodeFirework`), и нет нужды унифицировать. Меньше веток в общем `updateParticles`. |
| Зачем 1×1 заглушка-текстура? | Чтобы шейдер мог рендерить с первого кадра, пока реальная PNG ещё грузится по сети. Иначе sampler вернёт чёрный. |
| Зачем ограничивать `dt` сверху? | Если вкладка была в фоне, `dt` может стать секундой — за один шаг частица улетит на огромное расстояние, физика сломается. Кламп `min(0.033, dt)` защищает от этого. |
| Почему обход массива частиц с конца? | Чтобы при `splice(i, 1)` (удаление мёртвой частицы) индексы оставшихся элементов не сдвигались относительно итерации. |
