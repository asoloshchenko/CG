# Лаб 5 «Bump & Normal Mapping» — подробный разбор для защиты

Стек: **WebGL1** + чистый JS (без `gl-matrix` — матрицы вручную). GLSL ES 1.00 — поэтому `attribute/varying`, `texture2D`, `gl_FragColor` (а не `in/out` и `texture()` как в WebGL2). Файл — [main.js](main.js).

Главная задача: показать **два способа добавить «рельеф» без увеличения количества полигонов**:
1. **Bump mapping** на сфере (апельсин) — нормаль восстанавливается из карты высот (градиент яркости).
2. **Normal mapping** на второй сфере (камень) — готовая RGB-карта нормалей в tangent-space.

---

## 0. Зачем вообще bump/normal mapping

Геометрически сфера — гладкая. Если осветить её Фонгом, получится идеально гладкий шарик. Чтобы был виден рельеф (поры апельсина, трещины камня), есть три пути:
- **Реальная геометрия** — миллионы треугольников, медленно.
- **Bump mapping** — храним только **высоту** в grayscale-текстуре. В шейдере на лету вычисляем наклон поверхности (градиент высоты) и подменяем нормаль. Геометрия остаётся гладкой, но свет считает «как будто рельеф есть».
- **Normal mapping** — храним готовую **нормаль** прямо в RGB-текстуре (R→x, G→y, B→z, диапазон `[-1,1]` упакован в `[0,1]`). Шейдер читает её и использует вместо геометрической. Качественнее и дешевле, чем bump.

И в том, и в другом случае итоговая нормаль уходит в формулу освещения — и всё освещение «обманывается», что поверхность неровная.

## 1. Шейдеры (main.js:9–92)

### Вершинный
```glsl
vec4 worldPos = uModel * vec4(aPosition, 1.0);
vWorldPos = worldPos.xyz;
vNormal   = normalize(uNormalMatrix * aNormal);
vUV       = aUV;
vTangent  = normalize(uNormalMatrix * aTangent);
gl_Position = uProjection * uView * worldPos;
```
- `uNormalMatrix` (mat3) — для корректного поворота нормалей/тангенсов без translate-составляющей.
- Передаём во фрагмент: позицию в мире, нормаль, UV и **касательную** (tangent). Без касательной нельзя построить tangent-space.

### Фрагментный — ядро лабы
```glsl
vec3 normal    = normalize(vNormal);
vec3 tangent   = normalize(vTangent - normal * dot(vTangent, normal)); // Грэма–Шмидта
vec3 bitangent = normalize(cross(normal, tangent));
```
Здесь строится **базис TBN** (Tangent–Bitangent–Normal). Это локальная система координат в точке поверхности: T идёт «по U», B — «по V», N — наружу. Текстура нормалей всегда сделана **в этом базисе** (поэтому в normal map синий канал почти везде ~1: нормали смотрят «вверх», то есть из поверхности).

Tangent ортогонализуется к нормали методом **Грэма–Шмидта**: вычитаем из tangent проекцию на normal — гарантируем, что T⊥N.

Дальше — две ветки:

**Normal mapping (`useNormalMap > 0.5`):**
```glsl
vec3 mapNormal = texture2D(uNormalMap, vUV).rgb * 2.0 - 1.0;  // распаковка [0,1] → [-1,1]
mapNormal.xy *= uDetailStrength;
shadingNormal = normalize(T*mapNormal.x + B*mapNormal.y + N*mapNormal.z);
```
Прочитали нормаль из текстуры, распаковали из RGB, домножили xy на «силу» (z трогать нельзя — тогда нормаль уйдёт в плоскость), перевели обратно в мировое пространство `N_world = T·nx + B·ny + N·nz`.

**Bump mapping (иначе):**
```glsl
float hC = texture2D(uHeightMap, vUV).r;
float hU = texture2D(uHeightMap, vUV + vec2(uHeightTexel.x, 0.0)).r;
float hV = texture2D(uHeightMap, vUV + vec2(0.0, uHeightTexel.y)).r;
float dHdU = (hC - hU) * uDetailStrength;
float dHdV = (hC - hV) * uDetailStrength;
vec3 bumpNormalTangent = normalize(vec3(dHdU, dHdV, 1.0));
shadingNormal = normalize(T*... + B*... + N*...);
```
Берём высоту в текущей точке и в двух соседних (сдвиг на 1 пиксель текстуры — `uHeightTexel = 1/1024`). **Численная производная** по U и V даёт наклон поверхности. Из наклона собираем нормаль `(−dHdU, −dHdV, 1)` (здесь знак уже учтён порядком вычитания). Дальше — тот же перевод в мир.

**Освещение Фонга** поверх:
```glsl
vec3 ambient  = uAmbientColor * texColor.rgb;
float diff    = max(dot(shadingNormal, lightDir), 0.0);
vec3 diffuse  = diff * texColor.rgb * uLightColor;
float spec    = pow(max(dot(viewDir, reflectDir), 0.0), uShininess); // 48
vec3 specular = 0.35 * spec * uLightColor;
gl_FragColor  = vec4(ambient + diffuse + specular, texColor.a);
```

Ключевое: в формуле освещения используется не геометрическая `vNormal`, а **изменённая** `shadingNormal`. Это и есть «обман» — поверхность плоская, но свет видит её ребристой.

## 2. Математические утилиты (main.js:126–264)

- `multiplyMatrices(a, b)` — перемножение 4×4 матриц (column-major, как в OpenGL).
- `createPerspectiveMatrix(fov, aspect, near, far)` — классическая перспектива.
- `lookAt(out, eye, center, up)` — view-матрица камеры: строит ортонормированный базис (z = eye−center, x = up×z, y = z×x) и записывает в матрицу с обратным переводом начала координат.
- `createTransformMatrix(...)` — модельная матрица `T · Rz · Ry · Rx · S`, всё вручную через малые матрицы 4×4.
- `mat3FromMat4(out, m)` — выдирает верхний 3×3 блок (без translate) — для `uNormalMatrix`. Корректно при равномерном масштабе; иначе нужна `transpose(inverse(...))`.

## 3. `loadTexture(gl, url, flipY)` (main.js:270)

Шаги:
1. `createTexture` + сразу заливает **1×1 белый пиксель** через `texImage2D` — это «заглушка», чтобы до загрузки картинки можно было уже рендерить и не получить чёрный.
2. Возвращает `Promise`. В `image.onload`:
   - `pixelStorei(UNPACK_FLIP_Y_WEBGL, ...)` — нужно ли переворачивать по Y. Для апельсина — да (диффузка), для нормал-карты — **нет** (иначе сломаются Y-нормали).
   - `texImage2D` грузит реальную картинку.
   - Если размеры — **степени двойки** (`n & (n-1) === 0`) → `generateMipmap` + трилинейная фильтрация `LINEAR_MIPMAP_LINEAR`. Иначе — `CLAMP_TO_EDGE` + `LINEAR`. В WebGL1 без NPOT-расширения это обязательно: иначе текстура будет читаться как чёрная.
   - Magnification — всегда `LINEAR`.

## 4. `parseOBJ(text)` (main.js:327)

Парсит `v`, `vn`, `vt`, `f`. Поддерживает:
- Отрицательные индексы OBJ (`resolveIndex`).
- Триангуляцию веером.
- Дедупликацию по строке `"p/t/n"` через `Map`.
- **Авто-генерация UV** через `generateUVFromPosition` (`atan2/asin`) — сферическая проекция точки на UV. Спасает, если в OBJ нет `vt`.

Возвращает массивы `positions/normals/uvs/indices` + **тангенсы** через `buildTangents`.

## 5. `buildTangents(positions, normals, uvs, indices)` (main.js:403) — самое математически тяжёлое место

Тангенсов в OBJ нет — считаем сами. Для каждого треугольника:

Имеем рёбра `e1, e2` (в 3D) и соответствующие дельты UV `dUV1, dUV2`. Система:
```
e1 = dUV1.x · T + dUV1.y · B
e2 = dUV2.x · T + dUV2.y · B
```
В матричной форме `[e1 e2] = T·[dUV1 dUV2]`. Инвертируем:
```
det = dUV1.x * dUV2.y − dUV1.y * dUV2.x
T   = (dUV2.y * e1 − dUV1.y * e2) / det
```
Это даёт направление, вдоль которого U-координата растёт.

Дальше:
1. **Аккумулируем** T по всем треугольникам, к которым принадлежит вершина (одна вершина участвует в нескольких).
2. **Ортогонализуем** к нормали: `T -= N · (N·T)` — Грэма–Шмидта.
3. **Нормируем**. Если длина почти 0 (вырожденный случай, UV-шов и т.п.) — берём запасной вектор по правилу: если нормаль смотрит не вверх, `T = (-nz, 0, nx)`, иначе `(1, 0, 0)`.

Bitangent не сохраняем — он считается на лету в шейдере как `cross(N, T)`.

## 6. `createBuffer(gl, target, data, usage)` (main.js:499)

Тривиальный хелпер: `createBuffer → bindBuffer → bufferData`. Используется отдельно для positions, normals, uvs, tangents и indices.

## 7. `init()` (main.js:506) — оркестрация

1. **Грузит OBJ-сферу** (раньше грузил ещё и куб — теперь обе модели используют одну сферу).
2. Парсит → получает меш с тангенсами.
3. Создаёт программу, получает локации атрибутов и uniform'ов.
4. **Создаёт буферы один раз** в `sphereBuffers` и переиспользует для обеих моделей.
5. **Грузит 5 текстур параллельно** через `Promise.all`:
   - `food_0022_color_1k.jpg` — diffuse апельсина,
   - `food_0022_ao_1k.jpg` — height для bump-режима апельсина,
   - `Rock058_..._Color.png` — diffuse камня,
   - `Rock058_..._NormalGL.png` — normal map камня (в OpenGL-конвенции: Y+ вверх; есть ещё DirectX-вариант, где Y инвертирован),
   - `Rock058_..._Displacement.png` — height камня (для будущего использования).
6. Конструирует `models = { orange, rock }` — два «пресета» с разными текстурами, разным `useNormalMap` (0 или 1), своими диапазонами силы.
7. Камера статичная (`lookAt`), свет статичный.
8. Биндит samplers к юнитам: `uDiffuse → 0`, `uHeightMap → 1`, `uNormalMap → 2`.

## 8. `applyActiveModel()` (main.js:680)

Когда меняется активная модель:
1. `bindMeshBuffers` — настраивает `vertexAttribPointer` для всех 4 атрибутов + биндит EBO. Так как меш общий, по факту просто пере-настраивает указатели на те же буферы.
2. Биндит три текстуры активной модели в юниты 0/1/2.
3. Заливает `useNormalMap`, `detailStrength`, `heightTexel`.

## 9. Управление (main.js:718)

- **1** → orange (bump), **2** → rock (normal). Каждая модель помнит свою силу и диапазон.
- **↑ / ↓** → меняет `detailStrength` ± свой `strengthStep`, кламп к `[min, max]`. Заливает в uniform.

## 10. `render(timeMs)` (main.js:733)

Каждый кадр:
1. `resize()` — пересчитывает viewport с учётом `devicePixelRatio` (для HiDPI).
2. Проекция (FOV 45°).
3. Модельная: вращение вокруг Y со скоростью `t · 0.1`.
4. `mat3FromMat4` → uNormalMatrix.
5. `clear` + `drawElements(TRIANGLES, count, UNSIGNED_SHORT, 0)`.
6. `requestAnimationFrame(render)`.

## Что точно спросят на защите

| Вопрос | Ответ |
|---|---|
| Чем bump отличается от normal mapping? | Bump хранит **высоту** (grayscale), нормаль вычисляется в шейдере через градиент яркости. Normal хранит **готовую нормаль** в RGB. Normal качественнее и быстрее (3 чтения текстуры vs 1), bump гибче (одну height-карту можно использовать и для parallax, и для смещения вершин). |
| Что такое tangent-space и базис TBN? | Локальная система координат на поверхности: T — вдоль роста U, B — вдоль роста V, N — наружу. Нормали в normal map хранятся именно в TBN, поэтому почти везде B=1 (синий цвет). Нужно перевести их в мир: `N_world = T·nx + B·ny + N·nz`. |
| Зачем нужны касательные (tangents) и где их взять? | Чтобы построить TBN. В OBJ их нет — считаются на CPU из позиций и UV: решается линейная система через дифференциалы UV. |
| Что такое процесс Грэма–Шмидта в коде? | Ортогонализация: `T = normalize(T − N·(N·T))`. Гарантирует, что T строго перпендикулярна N, даже если усреднение тангенсов по треугольникам её слегка наклонило. |
| Почему `* 2.0 - 1.0` при чтении normal map? | Текстура хранит компоненты `[-1, 1]` упакованными в диапазон `[0, 1]` (unsigned byte). Распаковка: `nrm = rgb·2 − 1`. |
| Почему `mapNormal.xy *= strength`, а не `*=` целиком? | Если умножить и z, изменится только длина, не направление — после `normalize` эффект пропадёт. А масштабирование только xy реально «наклоняет» нормаль сильнее или слабее. |
| Как работает bump через градиент? | Численная производная высоты: `dH/dU ≈ (hC − hU)`, `dH/dV ≈ (hC − hV)`, шаг = 1 texel. Нормаль к плоскости с наклоном `(dH/dU, dH/dV)` есть `(-dH/dU, -dH/dV, 1)` (с точностью до знака). |
| Что такое NormalGL vs NormalDX? | Две конвенции normal map: в OpenGL ось Y направлена вверх, в DirectX — вниз. Если перепутать — рельеф будет «вывернут наизнанку» (свет приходит не с той стороны). Берём NormalGL под WebGL. |
| Зачем `UNPACK_FLIP_Y_WEBGL = false` для normal map? | Если отразить картинку по Y, инвертируется и Y-компонента нормалей внутри неё, и весь рельеф сломается. Для diffuse флип ок, для normal — нет. |
| Зачем 1×1 белый пиксель в loadTexture? | Текстура грузится асинхронно. Если попытаться рендерить до её готовности, sampler вернёт чёрный (или вообще ругнётся «texture not renderable»). Заглушка позволяет рисовать с первого же кадра. |
| Почему мипмапы только для NPOT? | В WebGL1 без расширения OES_texture_npot текстура не степень двойки **обязана** быть с `CLAMP_TO_EDGE` и фильтром без мипмапов, иначе она читается как чёрная. |
| Что делает uNormalMatrix? | Матрица 3×3 для преобразования нормалей (вращение/масштаб без translate). Строго правильно — `transpose(inverse(mat3(M)))`, у нас просто `mat3(M)` — работает при равномерном масштабе. |
| Чем varying отличается от in/out? | Это синтаксис GLSL ES 1.00 (WebGL1). Семантика та же: значение, выставленное в вершинном шейдере, интерполируется и читается во фрагментном. В WebGL2 их заменили на `out`/`in`. |
| Почему shininess = 48? | Степень в спекуляре Фонга: чем больше, тем «глянцевее» (узкий блик). 32–64 — типичная глянцевая поверхность. |
| Что такое uHeightTexel? | Размер одного texel'а в UV-координатах: `1 / width`. Нужен, чтобы знать, на сколько сдвинуть UV при чтении соседнего пикселя при численной производной высоты. |
